# Development Docs Overview

This folder keeps the planning view of the repository practical.
It answers:

What should we build next, in what order, and how do we keep the work reviewable?

## Why This Folder Exists
The project docs describe:
- product scope
- current system shape
- phased delivery direction
- the active backlog

This folder focuses on sequencing and backlog management rather than implementation detail.

## Planning Read Order
1. `docs/product-requirements-document.md`
2. `docs/plans.md`
3. `docs/frontend-designs.md`
4. `docs/backend-designs.md`
5. `docs/development/roadmap.md`
6. `docs/development/todo.md`

## Planning Rules
- Keep each issue small enough for one branch and one reviewable PR.
- Prefer vertical slices over broad platform work.
- Use the current repo baseline as the starting point.
- Prioritize the real personal workflow over speculative platform expansion.
- Update these docs when durable sequencing or scope decisions change.
